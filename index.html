Building the trust layer for privacy-preserving medical AI and research.

Solving the $300 Billion Medical AI Paradox

The Vision
Privacy as an Enabler, Not a Barrier

Universal Health Data Schemas (UHDS) provides a standardized framework for privacy-preserving health data exchange. We're building the missing interoperability layer that enables medical AI to scale while maintaining patient privacy.

By standardizing how we handle health data with built-in privacy mechanisms, we're solving the critical paradox: how to enable medical innovation while protecting patient confidentiality.

Core Capabilities

Privacy-by-Design

GDPR/HIPAA compliance built directly into data schemas with privacy classification for every field.

Federated Ready

Schemas designed for federated learning where models travel to data, not data to models.

Zero-Knowledge Queries

Verify data properties without exposing the data itself using cryptographic proofs.

Global Standards

Built on HL7 FHIR, ISO, and LOINC standards with extensions for modern AI workflows.

Example Schema: Patient Demographics

Here's what a UHDS schema looks like in practice:

{
  "$schema": "https://uhds.org/schemas/core/patient-demographics/v1.0.0",
  "title": "UHDS Patient Demographics",
  
  "properties": {
    "uhds_patient_id": {
      "type": "string",
      "format": "uuid",
      "uhds:privacy_class": "linkage",
      "description": "Pseudonymous identifier (never contains MRN/SSN)"
    },
    
    "birth_year": {
      "type": "integer",
      "uhds:privacy_class": "de-identified",
      "description": "Year of birth for de-identified datasets"
    },
    
    "postal_prefix": {
      "type": "string",
      "pattern": "^[A-Z0-9]{3}$",
      "uhds:privacy_class": "de-identified",
      "description": "First 3 chars of postal code for regional analysis"
    }
  },
  
  "required": ["uhds_patient_id", "birth_year"],
  
  "not": {
    "required": ["birth_date", "postal_code"],
    "description": "De-identified sets shouldn't have both exact DOB and full ZIP"
  }
}

Schema Design Principles

UHDS schemas follow core design principles that make them uniquely suited for privacy-preserving medical research:

Modular Design

Schemas are composable and extensible. Start with core demographics, add lab results, imaging data, etc.

Privacy Classification

Every field tagged with privacy class: PII, PHI, de-identified, linkage, metadata. Enforces proper handling.

De-identification Paths

Schemas provide both identified and de-identified versions of sensitive fields (birth_date → birth_year).

Provenance Tracking

Built-in provenance fields track data origin, transformations, and consent status for auditability.

Quick Start

Installation

# Install UHDS CLI and validation tools
npm install @uhds/cli @uhds/validator

# Or use the Python SDK
pip install uhds

# Validate a schema
uhds validate patient-demographics.schema.json

# Generate TypeScript/Java/Python types
uhds generate --lang typescript --out ./types/

Basic Usage Example

import { UHDSValidator } from '@uhds/validator';

// Load a UHDS schema
const schema = await loadSchema('patient-demographics.json');

// Create a validator instance
const validator = new UHDSValidator(schema);

// Validate data against schema
const data = {
  uhds_patient_id: 'a1b2c3d4-e5f6-7890-abcd-ef1234567890',
  birth_year: 1985,
  postal_prefix: '100',
  gender_identity: { code: 'female', system: 'http://hl7.org/fhir/administrative-gender' }
};

const result = await validator.validate(data);

if (result.valid) {
  console.log('Data is UHDS-compliant and privacy-safe');
  
  // Check privacy classification
  const privacyReport = validator.getPrivacyReport(data);
  console.log('Privacy class:', privacyReport.classification);
  
  // Get de-identified version
  const deidentified = validator.deidentify(data);
  console.log('De-identified data ready for sharing');
}
